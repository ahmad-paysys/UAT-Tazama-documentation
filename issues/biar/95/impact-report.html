Impact Study — Issue #95: BIAR Jupyter dashboards multi-tenancy interim fix

Issue: tazama-lf/biar#95
Study Date: 2026-07-13
Author: Ahmad Khalid
Repository: tazama-lf/biar
Scope: JupyterHub/notebooks/ — four erroring dashboards (Executive Overview, Case Management Trend, TMS Performance, Fraud Trend Analysis)

Summary

Four BIAR Jupyter dashboards either fail with a tenantId / AnalysisException error or silently render cross-tenant data because the shared helper load_tenant_hudi() matches a fixed snake_case column allowlist and one dashboard duplicates the filter with a hardcoded literal.

  • Track A — delete redundant Case Management Trend cell 4, extend the tenant-column allowlist to include camelCase (tenantId, txTenantId), and stop swallowing tenant-load errors in Executive Overview. Four notebooks, ~10 lines net change.
  • Track B — replace the hardcoded TENANT_FILTER_VALUE = "TAZAMA" with an ipywidgets dropdown in the same four notebooks so operators can switch tenants at runtime. Same four notebooks; the other three notebooks are deferred to a follow-up.

Confirmed Root Cause

All seven dashboard notebooks in biar/JupyterHub/notebooks/ define the same helper:

TENANT_FILTER_VALUE = "TAZAMA"
TENANT_FILTER_COLUMNS = ("tenant_id", "tx_tenant_id", "tenantid")

def load_tenant_hudi(path, **options):
    ...
    tenant_col = next((c for c in TENANT_FILTER_COLUMNS if c in df.columns), None)
    if tenant_col is None:
        raise ValueError(...)
    return df.filter(df[tenant_col] == TENANT_FILTER_VALUE)

Confirmed verbatim in:

  • Executive_Overview_Dashboard.ipynb cell 3
  • Case_Management_Trend_Dashboard.ipynb cell 1
  • TMS_Performance_Dashboard.ipynb cell 2
  • Fraud_Trend_Analysis_Dashboard.ipynb cell 1
  • Case_Tracking_Analysis_Dashboard.ipynb cell 2 not in fix scope
  • Fraud_Typology_Effectiveness_Dashboard.ipynb cell 2 not in fix scope
  • Anomaly_Detection_And_Rule_Calibration.ipynb cell 4 not in fix scope

Case Management Trend cell 4 re-applies a hardcoded literal:

alerts = alerts.filter(alerts.tenant_id == 'TAZAMA')
transactions = transactions.filter(transactions.tenant_id == 'TAZAMA')
cases = cases.filter(cases.tenant_id == 'TAZAMA')

Cell 1 already applies load_tenant_hudi() to all three DataFrames. When the underlying Hudi table uses a column named tenantid or tenantId instead of tenant_id, this line throws AnalysisException: cannot resolve 'tenant_id'.

Executive Overview cell 5 wraps the tenant load in try/except Exception as e: print(...). If load_tenant_hudi() raises because the allowlist doesn't match, downstream cells then hit NameError on alerts, cases, transactions.

Every notebook resolves the transactions table at ${WAREHOUSE_ROOT}/gold/transactions (plural). No notebook references a singular gold/transaction. The reporter's "Was it renamed transaction?" is a red herring: the string transactions appeared in the surfaced ValueError text, not in a renamed table.

What Track A fixes immediately:

  • Executive Overview, Case Management Trend, TMS Performance, Fraud Trend Analysis all execute end-to-end.
  • Tenant filter succeeds against Hudi tables that use camelCase (tenantId) column naming.
  • Tenant-load failures surface loudly instead of propagating as downstream NameError.

What Track A does NOT fix:

  • TENANT_FILTER_VALUE remains hardcoded to "TAZAMA" — operators still cannot switch tenants without editing the notebook.
  • Case_Tracking_Analysis, Fraud_Typology_Effectiveness, and Anomaly_Detection_And_Rule_Calibration are unchanged (deferred to follow-up).
  • The filter is still a display-only control, not a security boundary.

Track A is safe to ship in isolation.

Track B — Tenant selector widget

What Changes

Replace the hardcoded literal with an ipywidgets.Dropdown populated from a distinct-values scan of the tenant column at kernel startup. load_tenant_hudi() grows a tenant kwarg so the widget value is threaded explicitly. Applied to the same four notebooks as Track A.

Schema Impact

None — no new tables, no new columns, no migrations.

TMS_Performance_Dashboard.ipynbAdd tenant widget cell after cell 2; pass widget value to load_tenant_hudi.
Fraud_Trend_Analysis_Dashboard.ipynbAdd tenant widget cell after cell 1; pass widget value to load_tenant_hudi.

No new files. No enum sweeps. No reference sweeps.

Frontend Code Impact

None. ipywidgets are rendered natively by JupyterHub — the notebooks are the "frontend" here.

Acceptance Criteria

Track A
  • Case_Management_Trend_Dashboard.ipynb cell 4 removed.
  • TENANT_FILTER_COLUMNS in all four notebooks includes "tenantId" and "txTenantId".
  • Executive_Overview_Dashboard.ipynb cell 5 no longer swallows exceptions.
  • All four notebooks execute end-to-end without tenant errors.
  • Counts match an independent SQL query for the TAZAMA tenant.

Track B
  • Each of the four notebooks exposes an ipywidgets.Dropdown populated with distinct tenants.
  • Changing the dropdown and re-running downstream cells recomputes metrics for that tenant.
  • Default value is "TAZAMA".
  • No hardcoded "TAZAMA" literals remain in filter expressions in the four notebooks.

Recommended Sequencing
  1. Ship Track A as one PR against biar dev. No dependencies; unblocks the four erroring dashboards immediately.
  2. After Track A is merged and verified in the sandbox, ship Track B as a second PR against dev.
  3. Open a follow-up issue for the three notebooks out of scope (Case_Tracking_Analysis, Fraud_Typology_Effectiveness, Anomaly_Detection_And_Rule_Calibration) — same widget rollout, deferred per user direction.
  4. Longer-term: replace the interim filter with proper JupyterHub per-user-tenant authentication (belongs to the multi-tenancy epic, not this issue).
