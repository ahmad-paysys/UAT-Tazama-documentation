Issue: #225 — BIAR Bug: Case Management Trend Dashboard - no investigator data displayed
Date: 2026-07-10
Author: Ahmad Khalid
Repository: tazama-lf/case-management-system (with cross-repo scope: tazama-lf/biar, tazama-lf/Full-Stack-Docker-Tazama)

Summary

The Case Management Trend Dashboard's "Workload & Investigator Productivity" section shows raw user_id UUIDs instead of investigator display names. Everything upstream of NiFi is already in place on dev: CMS writes a Postgres row to cms_usernames on every successful login using generic OIDC claims (sub, name, tenant_id) — IAM-agnostic; a biar CmsUsernamesETL is registered and ready to write to gold/cms_usernames; the JupyterHub notebook already left-joins on user_id and falls back to UUIDs when no name is found. The single missing piece is a QueryDatabaseTableRecord NiFi processor for tazama_cms.cms_usernames. This is exactly the defect identified in biar #112 — which Sandy asked to be checked in the 2026-07-07 comment on #225.

Fixes are prepared: Full-Stack-Docker-Tazama PR #236 and biar PR #89 both OPEN and unmerged. Track A: merge them. No Track B is proposed — the login-driven staleness gap (users who never log in; renames-without-login; disabled users) cannot be closed inside CMS because the CMS repo must remain a plug-and-play IAM consumer with no IAM-vendor-specific code. Those cases are documented as known limitations.

2. Track A — Merge NiFi processor patch

What changes

Two files, both NiFi XML templates. The patch adds a QueryDatabaseTableRecord processor for tazama_cms.cms_usernames using the existing tazama_cms connection pool (b0b69311-8545-36e2), plus supporting RemoteProcessGroup table=cms_usernames entries so the batch is routed to the automation orchestrator with the correct dispatch key.

What Track A fixes immediately

  • gold/cms_usernames starts populating after the next NiFi poll interval.
  • Case Management Trend Dashboard resolves display names for anyone who has logged into CMS at least once since deployment.
  • Any future dashboard that joins on cms_usernames.user_id starts having data.

What Track A does not fix

  • Users who have never logged into CMS stay absent from cms_usernames.
  • Users whose display name is changed in IAM but who don't subsequently log in stay stale in the DLH.
  • Disabled users are never marked as such.
  • The architectural pattern (per-service identity mirroring in CMS) is unchanged.
Track A is safe to ship in isolation. It has zero source-code changes, no schema migration, and no coupling to open issues #214 or #220.

3. Known limitations (no Track B in CMS)

An event-driven identity sync — the natural follow-up that would fix the residual gaps below — is not proposed here because it would require IAM-vendor-specific code, and the CMS repository must remain a plug-and-play IAM consumer with no vendor coupling. The gaps are captured as known limitations of the login-driven mirror pattern.

LimitationCauseEffect on dashboardOperational workaround
User exists in IAM but has never logged into CMS cms_usernames is written only at login time Cases owned by this user show UUID, not name Have each active investigator log into CMS once after deployment
Name changed in IAM without subsequent CMS login Same Dashboard shows the previous name User re-logs in; or accept lag
User disabled in IAM No disable signal reaches cms_usernames Disabled user still appears in reports Manual filter downstream, or accept

If any of these ever needs to be closed, it must be done outside CMS — options include:

  • An external identity-sync service (in the deployment / ops repo) that talks to the IAM and writes directly to tazama_cms.cms_usernames, treating CMS's schema as an exported contract.
  • A NiFi processor that pulls from a standardized IAM export view/API into gold/cms_usernames, bypassing CMS entirely.
  • A generic REST intake endpoint on CMS (e.g. POST /internal/user-sync) fed by a deployment-side connector; CMS accepts the payload without knowing which IAM sent it.

None of these are in scope for #225 — the immediate report is fully resolved by Track A.
