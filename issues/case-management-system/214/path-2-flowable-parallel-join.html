Issue #214 — Path 2: Flowable Parallel Join

Issue: #214 — FRAUD_AND_AML container cases modelled as peer Case rows
Date: 2026-06-30
Path: Path 2 of 2 — Flowable-orchestrated parallel investigation with a coordinated combined SAR/STR close
Prerequisite: Path 1 — Minimal Flowable Change (Track A reporting fix should ship before this)

Executive Overview

Path 2 is the architecturally complete solution for clients who require that the combined closure of both FRAUD and AML investigations — including a single coordinated SAR/STR filing — be enforced and orchestrated by Flowable, not by application code or human convention.

Today, the coordination is done through a container Case row, a supervisor-only closure gate in application code, and hand-coded status propagation across four services. Path 2 replaces all of that with a new parent BPMN process that forks into two callActivity sub-processes (one per investigation), blocks at a parallel join gateway until both complete, then routes to a single combined SAR/STR filing task before ending. The container Case row is still removed (as in Path 1), but now Flowable itself becomes the coordinator.

Prerequisite: Path 2 builds directly on Path 1's Track B. The container Case row must be removed, the InvestigationGroup table introduced, and the five workaround layers deleted before the Flowable parallel process can take over coordination. Path 2 is additive on top of Path 1, not an alternative that skips it.

The Solution

New BPMN Process Design

A new BPMN process definition fraudAndAmlProcess is added alongside the existing caseManagementProcess. This new process is started when a FRAUD_AND_AML alert is triaged, taking alertId, tenantId, fraudCaseId, and amlCaseId as process variables.

The process contains:

  1. A parallel gateway (fork) — splits immediately into two branches
  2. Two callActivity elements — each calls caseManagementProcess as a sub-process, passing the respective caseId as the business key. Each sub-process runs its full lifecycle independently: investigation, supervisor approval, and individual SAR/STR if needed.
  3. A parallel join gateway — blocks until both callActivity sub-processes have reached their end events. Flowable maintains this join state durably across restarts and arbitrarily long waits.
  4. A combined SAR/STR filing user task — assigned to the compliance candidate group. This is a single task representing the joint regulatory filing that combines findings from both investigations.
  5. An end event — terminates the parent process, representing the closure of the full FRAUD_AND_AML investigation.

The existing caseManagementProcess is unchanged. Child processes invoked as callActivity use it without modification.

The isFraudNAMLGateway in the existing BPMN and the isFraudNAML variable are removed (clean-up) — they are no longer reached because FRAUD_AND_AML-typed containers no longer start a caseManagementProcess instance directly.

Application Code Changes

Path 2 includes all of Path 1 Track B — the container Case row is removed, all seven propagation blocks are deleted, the permission bypass is removed, and the closure gate is removed. Then Path 2 adds the following on top:

1. New BPMN file: fraud-and-aml-process.bpmn20.xml
Create the fraudAndAmlProcess definition with parallel fork, two callActivity elements calling caseManagementProcess, parallel join, combined SAR/STR user task, and end event. Deploy alongside the existing BPMN on startup.

2. FlowableService — new method handleFraudAndAmlCaseCreated
After both child cases are created in triage.service.ts, call this new method. It starts a fraudAndAmlProcess process instance, passing both fraudCaseId and amlCaseId as variables alongside alertId and tenantId. This is the process that Flowable keeps alive until both sub-processes complete.

3. triage.service.ts — capture both child case IDs
After creating the FRAUD and AML child cases, collect their case_id values and pass them to handleFraudAndAmlCaseCreated. Currently createCaseWithInvestigationTask returns { caseId, message } — this return value is used here.

4. flowable.service.ts — deploy second BPMN on init
Update deployBpmnProcess (or add a second deployment call) to also deploy fraud-and-aml-process.bpmn20.xml at module init time.

5. Combined SAR/STR task listener
A new Flowable task listener (or a Flowable event listener wired via the existing TaskEventListener) must handle the combined SAR/STR user task: assign it to the compliance group, surface it in the UI, and complete it when filing is done. The existing SAR/STR filing task in caseManagementProcess handles individual case filings; this new task handles the joint filing for FRAUD_AND_AML investigations.

6. Frontend — combined SAR/STR task display
The task list UI must surface the combined SAR/STR task as a distinct task type. It needs to show both the FRAUD case outcome and the AML case outcome together, so the compliance officer can file a single report that covers both. This requires a new task card variant and potentially a new modal.

Flowable Integration Points

Understanding how the CMS currently uses Flowable is essential to sizing Path 2 correctly.

Current model: One Case row → one caseManagementProcess instance, started via FlowableService.handleCaseCreated. The business key is caseId. The NestJS backend drives the process forward by calling handleTaskCompleted whenever a task transitions, which in turn calls Flowable's task completion API. Flowable evaluates the BPMN gateways and advances the token.

Path 2 model: One FRAUD_AND_AML alert → one fraudAndAmlProcess parent instance (business key: alertId) + two caseManagementProcess child instances (business keys: fraudCaseId and amlCaseId). The child instances are started by Flowable internally via callActivity, not by the NestJS service directly. The NestJS service still drives each child process forward through task completions as it does today.

The parallel join behaviour: Flowable natively supports parallel join. When both callActivity elements reach their end events, the join gateway fires automatically, creating the combined SAR/STR user task. The NestJS service does not need to poll or manually trigger the join — Flowable handles it. This is the key advantage over Path 1's application-code coordination.

Long-running process resilience: Investigations can take days or weeks. Flowable persists process state durably in its own database. The parallel join survives service restarts, container rescheduling, and database failovers. This is significantly more robust than the current application-code propagation, which loses state on restart.

Flowable API used:

  • POST /process-instances — start fraudAndAmlProcess
  • POST /deployments — deploy new BPMN (already wired in FlowableService.deployBpmnProcess)
  • Task completion continues to use the existing handleTaskCompleted path — child processes are driven exactly as single-type cases are today

Key design decisions in the BPMN:

  • Each callActivity calls the existing caseManagementProcess unchanged. The child processes are full investigation lifecycles including their own supervisor approval and individual SAR/STR filing steps. The combined SAR/STR in the parent is an additional, joint filing step on top of any individual filings — or the individual fileSarStr tasks can be suppressed for FRAUD_AND_AML child cases via a process variable if joint-only filing is the requirement.
  • The parallel join gateway waits indefinitely. Flowable stores the join state in its ACT_RU_EXECUTION table. This is robust across restarts.
  • The combined SAR/STR task receives both outcomes. The parent process variables include the finalOutcome from each child process, surfaced to the combined task so the compliance officer has full context.
  • The parent process does not have a Case row. It is a pure Flowable process entity, linked to the alert via alertId as business key and to the two child cases via InvestigationGroup.

BPMN XML sketch (key elements):

<process id="fraudAndAmlProcess" name="FRAUD and AML Combined Investigation" isExecutable="true">

  <startEvent id="start">
    <extensionElements>
      <flowable:formProperty id="alertId"     type="string" required="true"/>
      <flowable:formProperty id="tenantId"    type="string" required="true"/>
      <flowable:formProperty id="fraudCaseId" type="string" required="true"/>
      <flowable:formProperty id="amlCaseId"   type="string" required="true"/>
    </extensionElements>
  </startEvent>

  <parallelGateway id="parallelFork" name="Fork: start both investigations"/>

  <callActivity id="fraudInvestigation" name="FRAUD Investigation"
                calledElement="caseManagementProcess">
    <extensionElements>
      <flowable:in source="fraudCaseId" target="caseId"/>
      <flowable:in source="tenantId"    target="tenantId"/>
      <flowable:in sourceExpression="${'AUTOMATIC_SYSTEM'}" target="creationType"/>
      <flowable:in sourceExpression="${'false'}"            target="isFraudNAML"/>
      <flowable:out source="finalOutcome" target="fraudOutcome"/>
    </extensionElements>
  </callActivity>

  <callActivity id="amlInvestigation" name="AML Investigation"
                calledElement="caseManagementProcess">
    <extensionElements>
      <flowable:in source="amlCaseId" target="caseId"/>
      <flowable:in source="tenantId"  target="tenantId"/>
      <flowable:in sourceExpression="${'AUTOMATIC_SYSTEM'}" target="creationType"/>
      <flowable:in sourceExpression="${'false'}"            target="isFraudNAML"/>
      <flowable:out source="finalOutcome" target="amlOutcome"/>
    </extensionElements>
  </callActivity>

  <parallelGateway id="parallelJoin" name="Join: both investigations complete"/>

  <userTask id="combinedSarStr" name="Combined SAR/STR Filing"
            flowable:candidateGroups="compliance">
    <extensionElements>
      <flowable:formProperty id="sarStrAction" type="enum" required="true">
        <flowable:value id="inProgress" name="Keep Working"/>
        <flowable:value id="complete"   name="Complete Filing"/>
      </flowable:formProperty>
    </extensionElements>
  </userTask>

  <endEvent id="end" name="Combined Investigation Complete"/>

  <!-- Sequence flows -->
  <sequenceFlow sourceRef="start"            targetRef="parallelFork"/>
  <sequenceFlow sourceRef="parallelFork"     targetRef="fraudInvestigation"/>
  <sequenceFlow sourceRef="parallelFork"     targetRef="amlInvestigation"/>
  <sequenceFlow sourceRef="fraudInvestigation" targetRef="parallelJoin"/>
  <sequenceFlow sourceRef="amlInvestigation"   targetRef="parallelJoin"/>
  <sequenceFlow sourceRef="parallelJoin"     targetRef="combinedSarStr"/>
  <sequenceFlow sourceRef="combinedSarStr"   targetRef="sarStrGateway"/>
  <!-- gateway routes inProgress back to combinedSarStr, complete to end -->
  <sequenceFlow sourceRef="sarStrGateway"    targetRef="end">
    <conditionExpression>${sarStrAction == 'complete'}</conditionExpression>
  </sequenceFlow>
</process>

Tradeoffs & Pros/Cons
DimensionProsCons / Risks
Client requirement fit Fully satisfies a requirement that combined FRAUD+AML closure be orchestrated and enforced by the process engine, not by application code or human convention If the client's actual requirement is only "fix the data model and reports", Path 2 is over-engineering. Clarify before committing.
Process integrity Flowable guarantees the combined SAR/STR task cannot be skipped — it is a mandatory step in the parent process that fires only after both child processes end If either child process gets stuck (orphaned task, system error), the parent join never fires. Flowable's admin API must be used to diagnose and unblock, which requires ops tooling.
Long-running resilience Flowable persists process state durably. A parallel join waiting for two investigations that each take 3 weeks survives any number of restarts Flowable becomes a hard dependency for FRAUD_AND_AML closure. If Flowable is down, the combined task cannot be completed. Currently only tracked in application state (in-memory propagation blocks).
Audit trail The parent Flowable process provides a complete, engine-maintained audit of the fork, each sub-process completion, and the join — visible in Flowable's admin UI or history API Two audit trails now exist: application-level logging (NestJS LoggingOrchestrationService) and Flowable's own history. These must be kept consistent.
Delivery timeline Architecturally sound; no technical debt deferred Significantly more work than Path 1: new BPMN file, Flowable test environment validation, new frontend task card, new combined SAR/STR service. Estimate 8–12 days vs. 3–4 days for Path 1.
BPMN complexity callActivity is a well-supported Flowable pattern; parallel join is a standard BPMN 2.0 construct The BPMN must be tested in a Flowable test environment before production deployment. The callActivity variable mapping (flowable:in/flowable:out) requires careful validation — incorrect variable names silently produce wrong behaviour.
Individual vs. joint SAR/STR Explicit combined filing task makes the regulatory workflow unambiguous Decision required: do FRAUD and AML child processes also file individual SAR/STR reports, or is the combined parent task the only filing? The existing caseManagementProcess routes STATUS_82 outcomes to fileSarStr. If child processes should not file individually for FRAUD_AND_AML cases, they must be parameterised to skip that step, adding conditional logic to the existing BPMN.
Relationship to Path 1 Path 2 builds directly on Path 1 — all of Path 1's correctness gains apply, plus Flowable enforcement on top Cannot be delivered without Path 1 first completing. Any delay in Path 1 delays Path 2.

Acceptance Criteria

Inherited from Path 1 (all must pass):

  • Triaging a FRAUD_AND_AML alert creates exactly 2 Case rows (FRAUD and AML), both parent_id = null
  • An InvestigationGroup row is created with correct alert_id and tenant_id
  • Reports show accurate counts — no triple-counting
  • Status propagation between child cases does not occur
  • Standard permission checks apply to all cases

Path 2 specific:

  • Triaging a FRAUD_AND_AML alert starts a fraudAndAmlProcess Flowable process instance with fraudCaseId, amlCaseId, alertId, and tenantId variables correctly set
  • Closing only the FRAUD case does not trigger the combined SAR/STR task — the parallel join blocks until the AML case is also closed
  • When both FRAUD and AML cases close, the combined SAR/STR user task appears in the compliance group's task list exactly once
  • The combined SAR/STR task shows the FRAUD outcome and AML outcome from the respective child processes
  • Completing the combined SAR/STR task terminates the parent fraudAndAmlProcess instance
  • The parent Flowable process instance is visible in Flowable's admin/history API with the correct execution path
  • If the Flowable service restarts while the parallel join is waiting, the join resumes correctly and does not duplicate the combined task on the next completion event
  • Querying GET /cases?type=FRAUD_AND_AML returns zero results (no container row exists)
  • Zero Case rows with status = STATUS_84_COMPLETED after data migration
