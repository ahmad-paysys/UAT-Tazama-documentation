Issue #214 — Path 1: Minimal Flowable Change

Issue: #214 — FRAUD_AND_AML container cases modelled as peer Case rows
Date: 2026-06-30
Path: Path 1 of 2 — Remove container Case row; child cases close independently; no BPMN changes required
Companion: Path 2 — Flowable Parallel Join

Executive Overview

When a FRAUD_AND_AML alert is triaged, the system today creates three Case rows: one ownerless container (case_type = FRAUD_AND_AML) and two child investigations (FRAUD and AML). The container is a modelling error — it has no real owner, no investigator, and no genuine verdict. To hold it together, five layers of compensating code have accumulated: a synthetic status (STATUS_84_COMPLETED), a permission bypass, hand-coded parent↔child propagation across four services, a supervisor-only closure gate, and a frontend hardcode.

Path 1 removes the container Case row entirely and replaces it with a thin InvestigationGroup table record. The FRAUD and AML child cases become fully independent entities that each start their own Flowable process, follow the normal lifecycle, and close with their own STATUS_81/82/83 outcome. No changes to the BPMN file are required. Each case files its own SAR/STR independently.

The Solution

The solution ships in two independent tracks. Track A can merge and deploy today with zero risk; Track B is the structural re-model and requires coordinated release.

Track A — Fix Reporting Triple-Count Ship independently

One file changed: backend/src/modules/report/report.service.ts

Add a static helper constant to the class:

private static readonly LEAF_CASE_FILTER = {
  NOT: { case_type: CaseType.FRAUD_AND_AML },
} as const;

Spread this constant into buildCommonCaseFilters — the shared helper that every report query calls. This one addition propagates to all counts, breakdowns, trend queries, and workload reports. Two direct queries that bypass the helper (getInvestigatorWorkload and computeStatusDetails) are patched individually with the same spread.

Also remove STATUS_84_COMPLETED from the service's private CLOSED_STATUSES array and STATUS_DISTRIBUTION_MAP. These are local to the report service — the canonical constants in case.constants.ts are not touched in Track A.

What Track A fixes immediately:

  • Total case count: 1 FRAUD_AND_AML alert now contributes 2 to totals, not 3
  • Case-type breakdown: no longer shows a FRAUD_AND_AML bucket inflating totals
  • Closed totals: reconcile with STATUS_81/82/83 outcome breakdown
  • Average resolution time: no longer distorted by container rows closing near-instantly
  • Investigator workload: no longer attributes the ownerless container to the case creator

What Track A does not fix: the container row still exists and is still created for new alerts. The permission bypass, propagation code, and STATUS_84 remain load-bearing.

Track B — Full Re-model Coordinated release

Track B removes the container Case row and all five compensating layers. It ships as a single coordinated release — individual steps are not independently deployable because removing the permission bypass and closure gate requires the container row to already be absent from new data.

Step B1 — Add InvestigationGroup schema: A new, intentionally thin table: id, alert_id (unique), tenant_id, created_at. No status, no owner, no lifecycle. This is the correct abstraction for what the container was trying to be.

Step B2 — Stop creating the container Case row in triage.service.ts (two paths: manual at ~L372, auto at ~L612). Instead, upsert an InvestigationGroup record. Pass parentCaseId = null to both child case creation calls. The createCaseWithInvestigationTask signature in case-creation.service.ts widens from number to number | null.

Step B3 — Remove all seven parent↔child propagation blocks across four services:

  • task.service.ts: remove call site at L303 and entire promoteParentCaseToInProgress method (L318–353) — no remaining callers
  • task-lifecycle.service.ts: remove three if (updatedCase.parent_id) blocks at L65, L168, L273
  • case.service.ts: remove two blocks at L90 (suspend) and L229 (resume)
  • case-reopening.service.ts: remove two blocks at L68 and L244

Step B4 — Remove the permission bypass in case.repository.ts findCaseWithPermissionCheck (L185–196). The FRAUD_AND_AML OR-branch is deleted. FRAUD and AML child cases use the standard owner and task-assignee checks.

Step B5 — Remove the FRAUD_AND_AML closure gate in case-closure-approval.service.ts (L112–146). The isFraudAndAmlCase branch is deleted. All cases use the standard single-case closure path.

Step B6 — No BPMN changes required. See the Flowable section below.

Step B7 — Retire STATUS_84_COMPLETED (deferred until data migration validates zero rows): remove from case.constants.ts, case-enum.ts, and schema.prisma.

Step B8 — Data migration (see Data Migration section).

Step B9 — Frontend cleanup (see Frontend Impact section).

Role of Flowable in Path 1

Flowable is deeply integrated in the CMS — every Case row starts a Flowable process instance via FlowableService.handleCaseCreated. The BPMN uses the isFraudNAML variable to route FRAUD_AND_AML cases directly to the investigateCase user task, bypassing completeCaseCreation. Today, three cases → three Flowable process instances. The BPMN has no awareness of the parent↔child relationship; all coordination is application-code only.

Key insight: Path 1 requires zero changes to cms.bpmn20.xml. Child FRAUD and AML cases created after Track B will pass isFraudNAML = false to Flowable (they are standalone single-type cases). They follow the normal flowNotFraudNAML → completeCaseCreation path. The isFraudNAMLGateway, isFraudNAML form property, and flowIsFraudNAML sequence flow become dead code once no container row is created, but they do not need to be removed to ship the fix.

The FRAUD_AND_AML value in the completeCaseCreation user task's caseType form property (BPMN line 57) can also be removed in a follow-on BPMN cleanup, but this is cosmetic only.

What Flowable handles after Path 1:

  • Each FRAUD case: its own process instance, full lifecycle (investigation → supervisor approval → SAR/STR filing → close)
  • Each AML case: same, independently
  • No combined gate — Flowable does not know these two cases are related

What Flowable does not handle (accepted limitation of Path 1):

  • There is no Flowable-enforced gate requiring both FRAUD and AML to complete before a combined SAR/STR can be filed
  • Combined closure coordination is the responsibility of the investigator and supervisor at the application level, not enforced by the process engine

If the client's requirement is that combined closure must be orchestrated by Flowable, Path 1 does not satisfy it. That is the explicit scope of Path 2.

Data Migration

Run on staging with a full snapshot before production. Execute phases in order; validate each before proceeding.

-- Phase 1: Backfill InvestigationGroup for all existing FRAUD_AND_AML containers
INSERT INTO investigation_groups (alert_id, tenant_id, created_at)
SELECT a.alert_id, c.tenant_id, NOW()
FROM cases c
JOIN alerts a ON a.case_id = c.case_id
WHERE c.case_type = 'FRAUD_AND_AML'
  AND c.parent_id IS NULL
ON CONFLICT (alert_id) DO NOTHING;

-- Phase 2: Reclassify container Case rows (enables STATUS_84 retirement)
UPDATE cases
SET status = 'STATUS_83_CLOSED_INCONCLUSIVE'
WHERE case_type = 'FRAUD_AND_AML' AND parent_id IS NULL;

-- Validation gate (must be 0 before dropping the enum value)
SELECT COUNT(*) FROM cases WHERE status = 'STATUS_84_COMPLETED'; -- must be 0

-- Post Track-B validation
SELECT COUNT(*) FROM cases WHERE case_type = 'FRAUD_AND_AML';    -- must be 0
SELECT COUNT(*) FROM cases WHERE parent_id IS NOT NULL;            -- must be 0

Acceptance Criteria

Track A:

  • A FRAUD_AND_AML alert with both children closed contributes 2 to closedCases in reports, not 3
  • The case-type breakdown shows FRAUD and AML buckets only — no FRAUD_AND_AML bucket
  • closedCases equals the sum of STATUS_81 + STATUS_82 + STATUS_83 + STATUS_71 + STATUS_72 outcomes

Track B:

  • Triaging a FRAUD_AND_AML alert creates exactly 2 Case rows (FRAUD and AML), both with parent_id = null
  • An InvestigationGroup row is created with the correct alert_id and tenant_id
  • Assigning a task to the FRAUD case does not change the AML case status
  • Closing the FRAUD case does not trigger any update on the AML case
  • A user who is not the owner or task-assignee of a FRAUD case receives a 403
  • The close modal shows STATUS_81/82/83 outcomes only — no STATUS_84 option
  • Zero Case rows with status = STATUS_84_COMPLETED after data migration
  • Zero Case rows with case_type = FRAUD_AND_AML after data migration
