Impact Study on Moving Tables

Date: 2026-06-23
Scope: Tables proposed for removal from the configuration database
Services: admin-service, rule-studio (backend + database), connection-studio (backend), frms-coe-lib

Executive Overview

This document details the exact impact of every proposed table movement. For each table, it identifies:

  • Which HTTP endpoints are affected
  • Every file that must be changed across the stack
  • What specifically must change in each file

The movements are grouped into two operations:

  1. Move to Dev Tools DB — trs_nodes, trs_rule_flow, trs_rules, simulation_logs
  2. Move to Enrichment DB — job_history

Group 2: Move to Enrichment DB

Table: job_history

Affected endpoints:

ServiceMethodPath
admin-servicePOST/v1/admin/tcs/job/get/history/:offset/:limit
connection-studio backendPOST/job/history

Impact notes: The admin-service endpoint JOINs job_history with tcs_pull_jobs and tcs_push_jobs. After moving job_history to the Enrichment DB, this JOIN across databases is no longer possible in a single SQL statement. The endpoint must be refactored to:

  1. Fetch job history rows from Enrichment DB
  2. Enrich with job metadata via an in-process JOIN (load job configs from Configuration DB separately and merge in application code)

Files to change and what to change:

1. admin-service/src/index.ts
Add Database.ENRICHMENT to the CreateStorageManager call.
(Note: Database.ENRICHMENT already exists in frms-coe-lib's enum but is currently not connected in admin-service — verify whether admin-service already initialises enrichment.)

2. admin-service/src/services/database.logic.service.ts
Add a 'enrichment' case: databaseManager._enrichment.

3. admin-service/src/repositories/dataEnrichment/job.de.repository.ts (lines 242–299)
Change the handlePostExecuteSqlStatement('configuration', ...) call for the job_history SELECT to handlePostExecuteSqlStatement('enrichment', ...).
Refactor the JOIN query: The existing query JOINs job_history with tcs_pull_jobs and tcs_push_jobs. After the move, this cross-DB JOIN must be split:

  • Query 1 (Enrichment DB): SELECT * FROM job_history WHERE tenant_id = $1 ORDER BY created_at DESC LIMIT $2 OFFSET $3
  • Query 2 (Configuration DB): SELECT id, endpoint_name, source_type FROM tcs_pull_jobs WHERE id = ANY($1) and SELECT id, endpoint_name FROM tcs_push_jobs WHERE id = ANY($1) using the job IDs returned by Query 1
  • Merge in application service layer.

4. admin-service/src/services/job.logic.service.ts (or wherever history handler lives)
Update the getJobHistory() handler to perform the two-step fetch described above. Build the merged result set and return it in the same response shape to avoid breaking the API contract.

5. frms-coe-lib/src/builders/configurationBuilder.ts (lines 163–183)
The insertJobHistory() method currently runs against the configuration DB pool (this._configuration). Change it to run against the enrichment DB pool (this._enrichment). This means insertJobHistory must be moved out of ConfigurationDB and into EnrichmentDB.

6. frms-coe-lib/src/interfaces/database/ConfigurationDB.ts (lines 75–82)
Remove insertJobHistory() from the ConfigurationDB interface.

7. frms-coe-lib/src/interfaces/database/EnrichmentDB.ts
Add insertJobHistory(tenantId, jobId, counts, processedCounts, exception, jobType): Promise<void> to the EnrichmentDB interface.

8. frms-coe-lib/src/builders/enrichmentBuilder.ts
Implement insertJobHistory() using the _enrichment pool.

9. All callers of insertJobHistory() in the DEMS/enrichment processing code
Any service that calls db.insertJobHistory(...) currently resolves this through ConfigurationDB. After the move, it must resolve through EnrichmentDB. Locate all callers with: grep -rn "insertJobHistory" /home/ak/workspace/sim-final-week/ and update each call site to use the enrichment database manager handle.

10. connection-studio/backend/src/job/job.controller.ts (lines 94–103) and job.service.ts
No changes needed to connection-studio itself — it calls admin-service via HTTP. The response shape from admin-service's history endpoint must remain identical.

11. connection-studio/backend/tcs-init.sql (lines 149–158)
Remove the CREATE TABLE job_history DDL. It moves to Enrichment DB migration scripts.

12. Enrichment DB migration scripts
Add CREATE TABLE job_history (id SERIAL PK, tenant_id TEXT, job_id UUID, counts INTEGER, processed_counts INTEGER, exception TEXT, job_type TEXT, created_at TIMESTAMP DEFAULT NOW()).

13. Environment / deployment configuration (admin-service)
Confirm ENRICHMENT_DB_* environment variables are already present in admin-service config. If not, add them.
