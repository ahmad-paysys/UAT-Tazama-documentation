Impact Study — Issue #447

Title: Rule Studio pipeline builds against diverged, hardcoded feat-paysys branch of tazama-lf/rule-executer

Issue: #447
Study Date: 2026-07-03
Author: Ahmad Khalid
Repository: tazama-lf/rule-executer

1. Summary

Every rule deployed through Rule Studio clones the feat-paysys branch of tazama-lf/rule-executer because two workflow files in the tazama-lf/rule-studio-example template repo hardcode -b feat-paysys. The branch is 28 commits ahead and 35 commits behind dev, carrying hardcoded UAT IPs, disabled APM, a pre-release Paysys-internal library version, and raw console.log debug output — none of which should reach a production pipeline.

Track A is a two-line fix that retargets the two workflow files to dev. Track B reconciles the feat-paysys divergence into dev, cleans up debug artifacts, and deletes the branch.

2. Confirmed Root Cause
  • tazama-lf/rule-studio-example/.github/workflows/deploy.yml:39 on main/dev: git clone https://github.com/tazama-lf/rule-executer -b feat-paysys — confirmed by reading the file.
  • tazama-lf/rule-studio-example/.github/workflows/deploy-to-uat.yml:36: identical hardcoded clone — confirmed by reading the file.
  • git log --left-right origin/dev...origin/feat-paysys: 28 commits unique to feat-paysys, 35 commits unique to dev.
  • feat-paysys/package.json: frms-coe-lib at 0.0.1-psl.0 (private Paysys pre-release), frms-coe-startup-lib at 3.0.2-rc.5, no rule dependency, package version 3.0.0.
  • dev/package.json: frms-coe-lib at 8.2.0-rc.6, frms-coe-startup-lib at 3.1.0-rc.8, rule aliased to npm:@tazama-lf/rule-901@4.0.0-rc.6, package version 4.0.0-rc.2.
  • feat-paysys/Dockerfile: hardcoded RAW_HISTORY_DATABASE_HOST=10.10.80.18, PORT=15432, APM_ACTIVE=false, SERVER_URL=10.10.80.18:14222, RULE_NAME="rule-901", plain ARG GH_TOKEN (no BuildKit secret mount).
  • feat-paysys/src/controllers/execute.ts: 20+ [L##]-prefixed debug console.log/loggerService calls; databaseManager as any cast.
  • feat-paysys/src/controllers/rule.ts: new file present only on feat-paysys, containing five console.log("hello bhai…") statements. Never imported — execute.ts imports handleTransaction from the rule/lib package. This file is dead code.
  • feat-paysys/.husky/pre-commit: contains npx lint - staged (spaces around the dash) — broken hook.
  • feat-paysys/simple-rule2-test.js: 124-line orphan debug script, absent on dev.

Corrections vs prior write-ups:

  • The sed step at deploy.yml:66 is dead code but not a functional bug. Lines 108–111 of the same workflow perform npm pkg delete dependencies.rule followed by npm install --save-exact rule@npm:@$ORG/$RULE_NAME@latest, so the rule package is installed regardless of the sed outcome. informational
  • The hardcoded IPs in feat-paysys/Dockerfile are overridden at docker run time by the workflow's -e flags (deploy.yml lines 135–168, deploy-to-uat.yml lines 128–157). Fixing the Dockerfile removes a footgun for anyone running the image outside the pipeline; it does not unbreak the pipeline itself. informational
  • deploy-to-uat.yml is Paysys-specific by design — line 104 hardcodes the @psl-copilot npm scope. Retargeting the clone in this file does not unify the two workflows.
  • src/controllers/rule.ts on feat-paysys is orphaned dead code. Default action = delete.

Two workflow files in tazama-lf/rule-studio-example; one line each. No application code changes.

  • .github/workflows/deploy.yml — line 39: change -b feat-paysys to -b dev
  • .github/workflows/deploy-to-uat.yml — line 36: change -b feat-paysys to -b dev

What this fixes immediately
  • All future Rule Studio deploys clone dev instead of feat-paysys.
  • New rule containers pick up all 35 upstream dev commits (security fixes, dependency updates, bug fixes).
  • Debug logging, disabled APM, and the private 0.0.1-psl.0 library pin are no longer baked into new builds.

What this does not fix
  • Containers already deployed from feat-paysys remain running on diverged code until rebuilt.
  • The feat-paysys branch still exists and continues to diverge.
  • The 28 feat-paysys-only commits remain unreviewed and unmerged.
  • The dead sed pattern in deploy.yml:66 is not corrected by this track.
  • The @psl-copilot hardcoding in deploy-to-uat.yml:104 is unchanged — the two workflows remain non-unified by design.
  • No documentation of the build-time rule-injection contract is added.

Track A is safe to ship in isolation.

8. Recommended Sequencing
  1. Ship Track A immediately as a standalone PR into tazama-lf/rule-studio-example (both main and any branch consumed by Rule Studio bootstrap).
  2. Verify in parallel whether psl-copilot/rule-template also hardcodes -b feat-paysys; if so, apply the same one-line fix there.
  3. Audit the 28 feat-paysys-only commits — the residual risk after this study is the behavioural delta in execute.ts and any consumer of frms-coe-lib@0.0.1-psl.0 APIs that changed by 8.2.0-rc.6.
  4. Open Track B PR into dev on tazama-lf/rule-executer once the audit is complete. Merge as a single coordinated change.
  5. Rebuild and re-deploy all Rule Studio-generated rule containers after Track A and Track B are both merged.
  6. Delete feat-paysys from tazama-lf/rule-executer only after all deployed containers have been rebuilt and confirmed healthy.
