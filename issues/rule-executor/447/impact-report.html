Impact Study — Issue #447

Rule Studio pipeline builds against diverged, hardcoded feat-paysys branch

Issue: #447 — tazama-lf/rule-executer
Study Date: 2026-07-03
Author: Ahmad Khalid
Repository: tazama-lf/rule-executer & tazama-lf/rule-studio-example

Summary

Every rule deployed through Rule Studio clones feat-paysys — a diverged, unreviewed branch of rule-executer — because two workflow files in tazama-lf/rule-studio-example hardcode -b feat-paysys. The branch is 28 commits ahead and 35 commits behind dev, carrying hardcoded UAT IPs, disabled APM, a pinned Paysys-internal pre-release library, and raw console.log debug output baked into production images.

Track A is a two-line fix: change the clone target in both workflows to -b dev. Track B reconciles the feat-paysys divergence into dev and deletes the branch.

Track A — Switch clone target

What Changes

Two files in tazama-lf/rule-studio-example. One line each. No application code changes.

What this fixes immediately:

  • All future Rule Studio deploys clone dev — 35 upstream commits included
  • New containers get all upstream security/bug fixes (PRs #443, #445, #446)
  • Hardcoded IPs, disabled APM, and debug logging no longer baked into new builds
  • frms-coe-lib resolves from the public tazama-lf registry at the correct version

What this does not fix:

  • Containers already running from feat-paysys — remain on diverged code until rebuilt
  • feat-paysys branch still exists and still diverges
  • The 28 feat-paysys-only commits (including rule.ts) remain unreviewed
  • The dead sed pattern in deploy.yml line 66 — it works again only after Track B restores the placeholder

Track A is safe to ship in isolation.

Recommended Sequencing
  1. Ship Track A immediately as a standalone PR into tazama-lf/rule-studio-example — two lines, no risk.
  2. Audit the 28 feat-paysys-only commits in parallel — determine if rule.ts BaseMessage handling is required by the Studio pipeline or covered by dev's rule/lib import.
  3. Ship Track B as a reviewed PR into dev on tazama-lf/rule-executer once the audit is complete.
  4. Rebuild and re-deploy all Rule Studio–generated rule containers from dev after both tracks merge.
  5. Delete feat-paysys after all containers are confirmed healthy.
