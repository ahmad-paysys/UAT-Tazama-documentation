Summary

Issue #91 originally raised three findings against backend/: a missing Prisma schema, six jest suites that fail to load without a 32-byte ENCRYPTION_KEY, and a breaking @tazama-lf/tcs-lib 1.0.157-rc.0. In a follow-up the reporter retracted Finding #1 (backend does not import Prisma anywhere), and Finding #3 is handled through a tcs-lib pin.

The substantive work for this issue is Finding #2 — the test-env gap — plus removing the vestigial @prisma/client dependency. Track A is a config/docs fix; Track B moves the encryption-key validation off the module import path and hardens CI.

Confirmed Root Cause
  • src/utils/helpers.ts:16-22 throws at module-import when Buffer.from(ENCRYPTION_KEY, 'utf8').length !== 32. confirmed
  • backend/jest.config.ts:25-26 loads env via setupFiles: ['dotenv/config'], which only reads .env. .env is git-ignored; .env.example has a valid key but jest never reads it. confirmed
  • Six suites transitively import utils/helpers.ts via SftpService / NotificationService / JobService / DryRunService / SchedulerService. confirmed
  • Finding #1 (Prisma) is retracted: zero prisma matches inside src/. @prisma/client is declared in backend/package.json:41 but never imported. informational
  • Finding #3 (tcs-lib RC) is real but out of scope; coordinated via tazama-lf/tcs-lib#51 and a version pin at backend/package.json:46. informational

Track A — Test-Env Fix + Prisma Cleanup

What Changes
  • Add backend/.env.test (new file, ~5 lines) with a 32-byte ENCRYPTION_KEY and IV_LENGTH=16.
  • Update backend/jest.config.ts to load .env.test explicitly via a small test/jest.env.ts setup file.
  • Remove @prisma/client and prisma from backend/package.json.
  • Remove npx prisma generate from the root package.json backend:lint:eslint script.
  • Add a short section to backend/README.md documenting the test-env expectation.

What Track A fixes immediately

  • All 12 backend jest suites load from a clean checkout.
  • Partial-green runs that hid six missing suites can no longer happen.

What Track A does not fix

  • src/utils/helpers.ts still throws at import time — .env.test masks the symptom, not the design.
  • CI configuration (node-ci.yml) is not audited.
  • env.validation.ts still permits any string for ENCRYPTION_KEY.

Track A is safe to ship in isolation.

Track B — Reproducibility Hardening

What Changes

Track A plus source and CI hardening: move the 32-byte check off module-import, enforce length in the config validator, and add a clean-checkout smoke workflow.
