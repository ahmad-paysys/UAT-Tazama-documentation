Issue #220 — Divorce SLA Management from Prioritisation

Issue: #220 — Divorce SLA management from prioritisation
Date: 2026-06-30
Author: Ahmad Khalid
Repository: tazama-lf/case-management-system

Executive Overview

The Priority enum (NEW / URGENT / CRITICAL / BREACH) encodes two completely different things in a single database field: how inherently serious a case is, and how much SLA time has elapsed. A background cron job overwrites case.priority every hour using a time ratio — so any priority set by a supervisor or triage system based on genuine risk is silently replaced within 60 minutes. Investigators cannot trust the field. Filters return cases sorted by age, not severity. Reports flag old-but-low-risk cases as high priority.

The fix separates these into two orthogonal concepts: a stable severity label (LOW / MEDIUM / HIGH) set at triage and changed only by audited supervisor action, and a derived SLA state (ON_TRACK / AT_RISK / DUE_SOON / BREACHED) computed on read from a stored deadline — never written to the database by a background job.

The fix ships in two tracks. Track A is a nine-line deletion in one file and stops the cron overwrite immediately. Track B is the full structural separation: enum rename, SlaPolicy table, sla_due_at deadline column, rewritten cron job, priority actuator endpoint, and a sweep of ~12 backend and ~55 frontend files.

Confirmed Root Cause

The defect is confirmed in backend/src/modules/alert-priority/alert-priority.service.ts:

  • Line 43: const slaProgress = elapsedHours / slaHours — a pure time ratio, 0.0 to 1.0+
  • Line 45: const priorityScore = slaProgress — SLA time ratio assigned directly as the priority score input
  • Lines 46–54: priorityScore is fed into determinePriority — the same thresholds used at case creation with a genuine Tazama risk score — producing NEW / URGENT / CRITICAL / BREACH
  • Lines 65–72: The resulting value is written to case.priority via prisma.case.update

This runs on the cron defined in alert-priority.task.ts at 0 * * * * — every hour. Every tick, every open case has its priority field overwritten.

// alert-priority.service.ts — the conflation (lines 43–72)
const slaProgress = elapsedHours / slaHours;       // time ratio, not risk
const priorityScore = slaProgress;                 // SLA age masquerades as risk score
const priority = CasePriorityUtil.determinePriority(priorityScore);

if (alert.case_id) {
  await this.prisma.case.update({
    where: { case_id: alert.case_id },
    data: { priority },                            // overwrites triage-set value
  });
}

The schema already has sla_deadline DateTime? and sla_duration_hours Int? on the Case model (schema.prisma lines 135–136), confirming that deadline-based SLA infrastructure was partially anticipated but never connected to the cron job.

Track A — Stop the Cron Overwrite

Track A is a single-file change. It removes the nine lines in alert-priority.service.ts that write case.priority from the SLA time ratio. After this deletion, case.priority is written exactly once — at creation or triage via CasePriorityUtil.determinePriority with a genuine Tazama risk score — and never touched again by the background job.

The Change

One file, nine lines deleted: backend/src/modules/alert-priority/alert-priority.service.ts

Remove the if (alert.case_id) block (lines 65–72) entirely:

// DELETE these 9 lines from alert-priority.service.ts
if (alert.case_id) {
  await this.prisma.case.update({
    where: { case_id: alert.case_id },
    data: { priority },
  });
  this.logger.log(`Updated case ${alert.case_id} priority to ${priority}`);
}

The alert's own priority and priority_score fields continue to be updated by the same service — this is acceptable as an alert-level urgency display indicator and does not affect case filtering or reporting.

What Track A fixes immediately:

  • case.priority is now written exactly once: at creation or triage with a genuine Tazama risk score
  • Any manually-set or triage-set priority persists until the case is closed — it is no longer overwritten within the next cron cycle
  • Investigators and supervisors can trust the priority field

What Track A does not fix:

  • The Priority enum still uses NEW / URGENT / CRITICAL / BREACH vocabulary — time concepts even if the values are now stable
  • There is no visible SLA state — operators cannot see "HIGH severity, on-track for SLA"
  • Filtering by BREACH still means "was assigned that label at triage" rather than "is past deadline"
  • Escalation notifications have no deduplication and may re-fire unexpectedly
Track A is safe to ship in isolation. It creates a stable holding position while Track B is developed.

Track B — Full Separation of Priority and SLA

Track B implements the complete design: rename the enum, add sla_due_at and a SlaPolicy table, rewrite the cron job to derive and emit SLA state without writing to case.priority, stamp sla_due_at at case creation, and add an audited priority actuator endpoint. Track B must ship as a single coordinated release — the enum rename makes every partial state uncompilable.

The sla_due_at backfill uses created_at + 72h as a conservative default for all open cases. Closed cases are left with sla_due_at = NULL.

Frontend Impact

The frontend Priority const object is declared in frontend/src/features/alerts/types/triage.types.ts. Renaming the four values there cascades to all components that import and use them. The TypeScript compiler surfaces every stale reference as a type error, making the sweep mechanical once the type is updated.

Acceptance Criteria

Track A
  • A case that has been open for more than 24 hours does not have its priority field changed by the background cron job
  • A priority set by a supervisor during triage persists across multiple cron cycles
  • alert.priority and alert.priority_score continue to update correctly for alert-level display

Track B
  • Priority enum in schema and all code is LOW / MEDIUM / HIGH; no references to NEW, URGENT, CRITICAL, or BREACH remain in non-migration code
  • case.priority is never written by the SLA recalculation job
  • sla_due_at is populated for all cases at creation time
  • Changing priority via the actuator endpoint re-anchors sla_due_at on created_at + target(new_priority) — not on now
  • Priority change is recorded as a PriorityChanged audit event with actor, timestamp, old/new values, and reason
  • Escalation notifications are not re-fired for a (case_id, sla_state) pair that has already been notified in the current process lifetime
  • Workload report HIGH / MEDIUM / LOW priority buckets correctly reflect case severity, not SLA age
  • Filter by HIGH returns high-risk cases regardless of how long they have been open
  • All open cases have sla_due_at populated after migration
  • Zero cases or alerts have a priority value outside LOW / MEDIUM / HIGH after migration
